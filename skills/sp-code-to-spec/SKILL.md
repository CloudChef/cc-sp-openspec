# SP Code To Spec

## Core Rule

Use this optional skill to reverse-engineer an existing codebase into project documentation and current-state feature baselines. This is a bootstrap and analysis workflow, not a feature-change workflow. It must not write implementation code, create feature tasks, archive changes, or mark inferred behavior as a new approved requirement.

Generated durable feature documents must use one unified project documentation tree:

```text
docs/<project-name>/<module>/<feature>/
  <feature>.md
  spec/
    spec.md
    <function-point>-spec.md
  design/
    design.md
    <function-point>-design.md
  flow/
    flow.md
    <function-point>-flow.md
  other/
    <supporting-topic>.md
```

Do not generate `/sp-code-to-spec` current-state documents under `docs/constitutions/modules/` or `openspec/specs/`.

Generated documents default to Chinese unless the user explicitly requests English. Preserve code identifiers, paths, API routes, config keys, class names, and commands exactly when they appear in source references.

## When To Use

Use this skill when:

- A project already has code but lacks useful current-state project documentation.
- The user asks to generate initial OpenSpec context, project docs, rules, constitutions, or feature documents from existing code.
- The user asks to document current behavior before starting `/sp-brainstorm` or `/sp-goal`.
- The user asks to support code review or issue analysis by mapping changed code or a reported issue to existing feature docs.
- The team wants project-specific coding, configuration, testing, API, architecture, or business rules inferred from repeated code patterns.

Do not use it as the workflow for a specific new feature or bug fix. Use `/sp-brainstorm` or `CC-FixBug` for requested behavior changes.

## Inputs

- Existing source code, tests, build files, configs, scripts, migrations, API documents, README files, deployment documents, and runbooks.
- Existing `AGENTS.md`, `openspec/project.md`, `docs/rules/*.md`, `docs/constitutions/*.md`, `docs/wiki/*.md`, `docs/ai-context/source-index.md`, and existing `docs/<project-name>/` documents when present.
- User-provided scope such as repository root, project name, language, module, bounded context, capability, feature, code-review diff, or issue ID.

## Durable Outputs

Create or update only documentation, never production code:

- `docs/ai-context/source-index.md`
- `docs/ai-context/codebase-inventory.md`
- `openspec/project.md`
- `docs/<project-name>/readme.md` as the project documentation index when useful
- `docs/<project-name>/<module>/readme.md` as the module index when useful
- `docs/<project-name>/<module>/<feature>/<feature>.md`
- `docs/<project-name>/<module>/<feature>/spec/spec.md`
- `docs/<project-name>/<module>/<feature>/spec/<function-point>-spec.md` when the feature has multiple function points or the function point needs standalone detail
- `docs/<project-name>/<module>/<feature>/design/design.md`
- `docs/<project-name>/<module>/<feature>/design/<function-point>-design.md` when the feature has multiple function points or the function point needs standalone low-level design
- `docs/<project-name>/<module>/<feature>/flow/flow.md`
- `docs/<project-name>/<module>/<feature>/flow/<function-point>-flow.md` when the feature has multiple function points or the function point needs standalone flow detail
- `docs/<project-name>/<module>/<feature>/other/<supporting-topic>.md`
- `docs/rules/business-standards.md` when stable project-level business terms, policies, lifecycle states, or cross-feature rules are supported by repeated project patterns or explicit user confirmation
- `docs/rules/<language>-code-standards.md` or `docs/rules/<language>-<runtime>-code-standards.md`
- `docs/rules/<project-rule>.md`
- `docs/constitutions/<area>.md`

Optional process outputs:

- `.agent/workdir/sp-openspec/bootstrap/code-to-spec-review.md`
- `.agent/workdir/sp-openspec/code-review/<review-id>/code-to-spec-review.md`
- `.agent/workdir/sp-openspec/issue-analysis/<issue-id>/code-to-spec-analysis.md`

## Documentation Structure Rules

- Derive `<project-name>` from the repository root, product name, existing docs, build metadata, or explicit user input. Use stable kebab-case.
- Documentation hierarchy is product/project -> module -> feature -> function point:
  - Identify modules inside the product or project.
  - Identify features inside each module.
  - Identify function points inside each feature.
- All current-state feature documents generated by this workflow belong under `docs/<project-name>/`.
- Use real module names for `<module>` and real user/business feature names for `<feature>`.
- Do not use generic names, placeholders, sequence numbers, or vague buckets such as `module-1`, `feature-1`, `capability-001`, `common`, `misc`, `general`, `default`, or `design`.
- A feature directory must describe one independently understandable feature. Split features when they have different public entry points, actors, state flows, permission semantics, data side effects, or test/verification entry points.
- Within each feature, identify concrete function points before drafting docs. A function point is a distinct user-visible or system-visible operation inside the feature, such as create, update, approve, import, sync, calculate, validate, query, export, notify, retry, or schedule.
- `<feature>.md` is the feature index and must list the feature's function points with links to their spec/design/flow details.
- `spec/spec.md`, `design/design.md`, and `flow/flow.md` are feature-level overview/index documents. They must not be the only detail documents when the feature contains multiple function points with different entry points, data flows, classes, method signatures, IO, or verification paths.
- For a simple feature with one function point, the function-point detail MAY live inside `spec/spec.md`, `design/design.md`, and `flow/flow.md` under a clearly named function-point section.
- For a feature with multiple function points, create function-point detail files in `spec/`, `design/`, and `flow/` using real function-point names, for example `<function-point>-spec.md`, `<function-point>-design.md`, and `<function-point>-flow.md`.
- If a module has many features, create one child directory per feature. Do not flatten a large module into one catch-all document.
- `<feature>.md` explains what the feature is and links to `spec/`, `design/`, `flow/`, and `other/`.
- `spec/spec.md` summarizes current externally observable behavior and links to function-point specs when present.
- `design/design.md` summarizes current technical realization, feature-level source mapping, shared class/module responsibilities, and links to function-point low-level design files when present.
- `flow/flow.md` summarizes current business/process flow and links to function-point flows when present.
- `other/` is for supporting notes such as source maps, API notes, data notes, config notes, diagrams, glossary, and owner questions.

## What Not To Generate

Do not generate or require old fixed matrix, evidence, or owner-question section templates for `/sp-code-to-spec`.

Use simpler project documentation instead. Keep source references close to the relevant claim, and put unclear or owner-dependent behavior under a concise "待确认事项" section only when needed.

## Workflow

1. Confirm scope from the command or user prompt. If the scope is unclear and multiple roots, project names, or modules are plausible, ask one concise clarification question.
2. Read repository entry points first: `AGENTS.md`, README files, build files, package manifests, service configs, deployment files, test configs, and existing `openspec/` / `docs/` artifacts.
3. Build or update a codebase inventory:
   - Languages, frameworks, runtime versions, package managers
   - Project name and module list
   - Public entry points, APIs, CLIs, jobs, scheduled tasks, event handlers
   - Controllers, services, repositories, clients, integrations, shared modules
   - Data stores, migrations, models, caches, queues, external services
   - Test layout, fixture strategy, E2E capability, coverage tooling
   - Logging, traceability, auth, security, config, and deployment patterns
4. Build a Project / Module / Feature / Language map before drafting files:
   - Project name and documentation root
   - Modules identified inside the product/project, with module name, source roots, language/runtime, public entry points, and source references
   - Features identified inside each module, with feature name, actor/client, business purpose, owning module, supporting modules, entry points, outputs, side effects, and source references
   - Function points inside each feature, with function-point name, trigger, entry point, owning classes/modules, data flow, IO, verification entry, and source references
   - Language/runtime standards needed for each module
   - Cross-module dependencies and data ownership boundaries
5. Apply feature independence checks:
   - Split features by public entry point, actor/client, business goal, permission boundary, state flow, data side effect, async/external IO behavior, or independent verification entry.
   - Keep closely coupled feature variants together only when they share the same business goal, source boundary, and verification entry.
   - Do not mix unrelated features in one `<feature>.md`, `spec.md`, `design.md`, or `flow.md`.
6. Apply function-point split checks inside each feature:
   - Identify all function points visible from controllers, routes, CLIs, jobs, event handlers, UI actions, service operations, tests, workflow states, or documented operations.
   - Split function-point details when they have different entry points, class sets, data flows, persistence behavior, external IO, async behavior, method signatures, or verification paths.
   - Do not create generic or numbered function-point files. File names must use real function-point names, such as `create-order-design.md`, not `function-1-design.md`.
   - Keep shared feature-level concepts in `spec/spec.md`, `design/design.md`, and `flow/flow.md`; keep function-point-specific behavior, low-level design, and process details in function-point sections or files.
7. Draft the feature `<feature>.md` in the conversation before writing files. It should include:
   - Feature purpose
   - Actors/clients
   - Entry points
   - Function point list with links to function-point spec/design/flow details
   - Main current behavior
   - Important variants or special cases
   - Data/state side effects
   - Verification or test entry points when visible
   - Links to `spec/`, `design/`, `flow/`, and `other/`
8. Draft `spec/spec.md` and any needed function-point spec files in the conversation before writing files. They should describe current observable behavior with:
   - Purpose
   - Scope
   - Actors/clients
   - Entry points
   - Function point index and links when function-point files exist
   - Current behavior
   - Business rules
   - Inputs and outputs
   - Important variants and error cases
   - Source references
   - 待确认事项, only when behavior is unclear or owner-dependent
9. Draft `design/design.md` and any needed function-point design files in the conversation before writing files. `design/design.md` should describe feature-level design and shared implementation. Function-point design files should contain the low-level design for each function point. Include:
   - Source roots and key files
   - Public entry points and internal boundaries
   - Function point index and links when function-point files exist
   - Controller/service/repository/client/job responsibilities when relevant
   - All feature-related classes/modules for each function point
   - Low-level class/module responsibilities for each function point
   - Key method signatures for each function point, including parameter names, concrete types/schemas, required/optional status, defaults, validation, return values, and source references
   - Data flow for each function point from entry point through services, repositories, clients, events/jobs, persistence/cache/queue, and response/output
   - Data ownership, schema/model/cache/queue usage
   - External integrations and failure behavior
   - Configuration keys and defaults
   - Logging, `trace_id`, and security behavior
   - Test coverage and runnable verification paths
   - Known risks and owner decisions needed
10. Draft `flow/flow.md` and any needed function-point flow files in the conversation before writing files. They should describe:
   - Trigger and preconditions
   - Function point index and links when function-point files exist
   - Main process path
   - Important alternative or error paths
   - Data/state changes
   - External IO or async behavior when present
   - Final observable result
   - Optional Mermaid diagram when it improves readability
11. Draft project business standards and language/runtime rules in the conversation before writing files. Project-level business rules must define reusable terms, lifecycle states, policy decisions, invariants, and cross-feature behavior only when repeated project patterns or explicit user confirmation support them.
12. Run main-process review before writing files:
   - Source mapping check: every behavior/rule claim has a project-owned source reference or explicit user confirmation.
   - Overreach check: no invented desired behavior, future scope, compatibility promise, or business rule is promoted from weak signals.
   - Project/module/feature split check: documents follow `docs/<project-name>/<module>/<feature>/`.
   - Feature independence check: each feature directory is independently understandable and not a catch-all.
   - Function-point decomposition check: every visible operation inside the feature is listed as a function point, and function-point details are split when entry points, classes, data flows, method signatures, IO, or verification paths differ.
   - Low-level design check: function-point design covers related classes/modules, method signatures, parameter definitions, return values, data flow, IO, and verification paths when visible in source.
   - Document naming check: project, module, and feature paths use real names without generic names or sequence numbers.
   - Function-point naming check: function-point files use real function-point names and not generic names or sequence numbers.
   - Section check: no old fixed matrix, evidence, or owner-question section templates are generated.
   - Language/runtime split check: each generated code standard applies to one language/runtime family and names the modules it covers.
   - Conflict check: new drafts do not contradict existing `AGENTS.md`, `openspec/project.md`, rules, constitutions, source-index entries, or existing docs.
   - Maintenance check: generated docs are concise enough for later `/sp-brainstorm`, `/sp-spec`, `/sp-tasks`, `/sp-impl`, code review, and issue analysis.
13. Review generated drafts in the main thread only. Do not start independent review threads for `/sp-code-to-spec`.
14. Confirm findings, fix confirmed issues, and re-review affected drafts until zero unresolved blocking findings.
15. Ask the user to confirm the reviewed drafts before file creation or updates. If the user requests changes, revise and re-run affected review before writing.
16. After confirmation, create or update the agreed documents and record review notes in `.agent/workdir/sp-openspec/bootstrap/code-to-spec-review.md` when process notes are useful.

## Code Review Support Mode

Use this mode when the user asks `/sp-code-to-spec` to support a code review.

Rules:

- Do not update durable docs by default.
- Map changed files to `docs/<project-name>/<module>/<feature>/` documents, `docs/rules/*.md`, and source references.
- Identify whether the diff matches current documented behavior, changes documented behavior, or exposes stale/missing docs.
- Return findings in a concise review format with severity, changed file/path, related feature doc, issue, impact, and recommended next action.
- If docs are stale, record a documentation update recommendation. Do not silently rewrite durable docs unless the user confirms.
- Write optional process notes to `.agent/workdir/sp-openspec/code-review/<review-id>/code-to-spec-review.md` when useful.

## Issue Analysis Support Mode

Use this mode when the user asks `/sp-code-to-spec` to analyze an issue, incident, Jira, bug report, or support case against existing code.

Rules:

- Do not update durable docs by default.
- Identify the reported entry point, current observed behavior, expected behavior source, related module, related feature, source references, and likely affected flow.
- Compare the issue with existing `docs/<project-name>/<module>/<feature>/` docs and source code.
- Classify the issue as likely bug, expected current behavior, unclear owner decision, stale documentation, environment/config issue, or insufficient data.
- Recommend the next workflow: `CC-FixBug`, `/sp-brainstorm`, documentation update, or more context collection.
- Write optional process notes to `.agent/workdir/sp-openspec/issue-analysis/<issue-id>/code-to-spec-analysis.md` when useful.

## Spec Document Guidance

Store current-state specs under:

```text
docs/<project-name>/<module>/<feature>/spec/spec.md
docs/<project-name>/<module>/<feature>/spec/<function-point>-spec.md
```

Recommended sections:

```md
# <Feature Name> Spec

## Purpose

## Scope

## Actors / Clients

## Entry Points

## Function Points

## Current Behavior

## Business Rules

## Inputs / Outputs

## Variants And Error Cases

## Source References

## 待确认事项
```

Use `SHALL` only when current behavior is strongly supported by project-owned sources or explicit user confirmation. If behavior is unclear, put it under `待确认事项` instead of treating it as approved.

## Rule Guidance

Rules must describe reusable project standards, not one feature's accident.

Generate or update `docs/rules/business-standards.md` when repeated project patterns or explicit user confirmation support stable project-level business definitions. It should capture reusable business glossary terms, domain object definitions, lifecycle/status semantics, cross-feature policies, invariants, approval/permission meanings, and source-of-truth rules.

For multi-language projects, generate or update separate language/runtime rule files. Examples:

- `docs/rules/java-code-standards.md`
- `docs/rules/python-code-standards.md`
- `docs/rules/typescript-react-code-standards.md`
- `docs/rules/go-cli-code-standards.md`
- `docs/rules/sql-migration-standards.md`

Each language/runtime rule file must state which modules it applies to. Shared cross-language standards, such as logging, encoding, API, security, and testing, should remain in shared rule or standard files.

Do not generate rules from:

- A single questionable implementation
- A single feature branch that has not been repeated or confirmed as a project rule
- Dead code, generated code, vendored dependencies, third-party examples, snapshots, or minified assets
- Provider-specific folder names unless the project explicitly treats them as a general architecture rule
- Temporary migration logic or backwards-compatibility code unless project docs require it

## Rules

- This workflow is optional and may run before normal `/sp-brainstorm` / `/sp-goal` work.
- Do not insert `/sp-code-to-spec` into the required `/sp-goal` phase sequence.
- Do not create `openspec/changes/<change-id>/` artifacts unless the user explicitly asks for a normal change workflow.
- Do not write production code, tests, migrations, or configs.
- Current-state feature docs belong under `docs/<project-name>/<module>/<feature>/`, not `docs/constitutions/modules/` and not `openspec/specs/`.
- Every feature directory must include `<feature>.md`, `spec/`, `design/`, and `flow/`; create `other/` when supporting notes are needed.
- Documentation hierarchy is product/project -> module -> feature -> function point. Identify modules inside the product/project, identify features inside each module, and identify function points inside each feature before drafting documents.
- Every feature must identify its function points. Simple single-function-point features may keep detail in `spec/spec.md`, `design/design.md`, and `flow/flow.md`; multi-function-point features must create function-point detail files under `spec/`, `design/`, and `flow/` using real function-point names.
- Multi-module projects must generate module-scoped and feature-scoped docs instead of one catch-all spec/design.
- Multi-language projects must keep language/runtime code standards separate and must record which modules each rule file applies to.
- Every generated project/module/feature path must use real project, module, and feature names.
- Do not generate generic or numbered paths such as `module-1`, `feature-1`, `capability-001`, `common`, `misc`, `general`, `default`, or `design`.
- Do not require or generate old fixed matrix, evidence, or owner-question section templates for this workflow.
- Code identifiers and source paths are allowed only in source references, source mapping, design baselines, or rule provenance.
- Generate or update `docs/rules/business-standards.md` only when stable project-level business terminology, lifecycle states, policies, invariants, or cross-feature rules are supported by repeated project patterns or explicit user confirmation.
- Do not create broad rules from code that appears accidental, duplicated, legacy-only, or module-specific.
- Prefer updating `docs/ai-context/source-index.md` so later workflows know which generated docs/rules/standards to read.
- Record conflicts and owner-dependent behavior explicitly instead of guessing.
- If generated docs would conflict with existing approved specs or rules, stop and ask the user how to resolve the conflict.
