# CC Commit

## Core Rule

Use this skill to create a local bug-fix commit and submit it to Gerrit for review. This is the required final step for `cc-fixbug` after implementation, review, deployment, and post-deploy verification are complete.

The commit and Gerrit review must describe a Jira bug fix. The commit message must be written in Chinese and must include the bug fix requirement, bug ID, Jira ID, bug name, root cause, solution, modified points, test evidence, deployment evidence, and review evidence.

Do not commit `.agent/workdir/` process artifacts.

## Inputs

- Jira ID, such as `ABC-123`
- Bug ID when different from the Jira ID
- Bug name/title
- Root cause of the bug, including the failed assumption or broken code/data path that produced the user-visible issue, backed by concrete evidence when available
- Workdir evidence from `.agent/workdir/cc-fixbug/<jira-id>/`
- Code/test/project file changes that implement the fix
- Deployment evidence from `.agent/workdir/cc-fixbug/<jira-id>/cc-deploy.md`
- Gerrit remote and target branch

If `cc-commit` is invoked directly and the Jira ID, bug name, or workdir evidence is missing, inspect the current branch, recent workdir folders, and staged/unstaged diff. Ask one concise question only when the bug cannot be identified safely.

## Preconditions

Before committing:

1. Read `.agent/workdir/cc-fixbug/<jira-id>/brainstorm.md`, `spec.md`, `design.md`, `tasks.md`, `task-reviews.md`, `review.md`, and `cc-deploy.md` when present.
2. Confirm all implementation/review findings are closed or explicitly false-positive with evidence.
3. Confirm the root cause has been identified and is stated concretely enough for reviewers to understand why the bug happened. Prefer citing concrete evidence such as request parameters, logs, timestamps, stack lines, database state, or reproducible steps.
4. Confirm tests and standalone bug-entry verification are recorded.
5. Confirm `cc-deploy.md` records successful remote deployment and post-deploy verification, or records an explicit user/project-approved reason deployment is not applicable.
6. Confirm the working tree does not contain unrelated changes. If unrelated changes exist, stage only the bug-fix files.
7. Confirm `.agent/workdir/` files are not staged.
8. Confirm Gerrit submit target:
   - Prefer a configured `gerrit` remote.
   - Otherwise use the project-approved review remote.
   - If only `origin` exists and it is the Gerrit remote, use `origin`.
   - Determine target branch from the current branch upstream, project docs, or user-provided target.
9. Confirm Gerrit `commit-msg` hook or project-approved Change-Id mechanism is available when the Gerrit server requires `Change-Id`.

If the Gerrit target or required Change-Id mechanism cannot be determined, stop before commit/push and report the blocker.

## Commit Message Format

Use a concise Chinese subject and a structured Chinese body. Except for fixed technical identifiers such as Jira ID, Bug ID, Gerrit `Change-Id`, commands, file paths, API paths, configuration keys, class names, method names, branch names, and remote names, the commit subject, section labels, and body content must be written in Chinese.

```text
修复：<Bug 名称>

Bug 修复：<Bug 名称>
Bug ID：<bug-id>
Jira: <jira-id>
Bug 名称：<Bug 名称>

根因：
- <说明 Bug 发生的具体原因，例如错误假设、缺失校验、过期数据路径、并发竞争、配置不一致或依赖行为变化>
- <说明证明根因的证据，例如请求参数、日志、时间戳、数据库记录、堆栈行或复现步骤>

修复方案：
- <修复策略摘要>

修改点：
- <修改的模块、文件、函数区域以及修改原因>

测试验证：
- <测试或验证命令及结果>

部署验证：
- <目标环境、受影响服务、部署结果和部署后验证>

Review 结论：
- <Review 证据摘要，确认没有未关闭 finding>

Change-Id: <generated by Gerrit commit-msg hook when required>
```

Rules:

- The whole commit message must be Chinese by default. Do not use English section labels such as `Root-Cause`, `Solution`, `Modified-Points`, `Tests`, `Deploy`, or `Review`.
- `Bug ID` is required. If the project has no separate bug ID, use the Jira ID.
- `Jira` is required and must contain the Jira ID.
- `Bug 名称` is required and must match the Jira title/name or user-confirmed title.
- `根因` is required and must explain the concrete cause of the bug, not only restate the symptom. When evidence exists, include the evidence in the commit message.
- `修复方案` is required and must state the actual fix strategy.
- `修改点` is required and must list the meaningful code/test/config changes.
- `测试验证` is required and must list real commands or verification evidence.
- `部署验证` is required and must list the target environment, deployed services, result, and post-deploy verification, or the approved not-applicable reason.
- `Review 结论` is required and must state whether findings are closed.
- Keep the body complete but not excessively detailed.

## Gerrit Submission

After creating the local commit:

1. Verify the final commit message contains the required Chinese fields and does not use English section labels except fixed technical identifiers.
2. Verify `.agent/workdir/` artifacts are not included in the commit.
3. Submit to Gerrit review, not a normal branch push:

```text
git push <gerrit-remote> HEAD:refs/for/<target-branch>
```

4. If the project requires review labels, topics, reviewers, or hashtags, use the project-documented Gerrit push options.
5. Record the commit hash, target branch, Gerrit remote, push command, and Gerrit review URL or push output in `.agent/workdir/cc-fixbug/<jira-id>/cc-commit.md` when the workdir exists.

## Do Not

- Do not push directly to `main`, `master`, or a normal remote branch for this workflow.
- Do not submit without Jira ID, bug ID, bug name, root cause, solution, modified points, tests, deployment, and review evidence in the commit message.
- Do not submit without deployment evidence or an approved deployment-not-applicable record.
- Do not commit `.agent/workdir/` process evidence.
- Do not include unrelated changes.
- Do not mark Gerrit submission complete if the push fails.
- Do not ask for another user confirmation after `cc-fixbug` brainstorm confirmation unless the Gerrit target, Jira identity, deployment environment details, or credentials are missing.
