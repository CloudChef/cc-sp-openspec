# Encoding Standards

> 默认语言：除非用户明确要求英文，本文档的标题、章节内容和说明均使用中文；编码名、配置键、命令和代码标识符保持原文。

这些规则用于防止 AI 生成或修改代码、注释、配置、测试数据和文档时出现乱码、错误转码或不可读字符。

## ENC-001: 统一编码

生成或修改的文本文件 MUST 使用目标项目认可的编码；没有项目特殊要求时 MUST 使用 UTF-8。

- 新增源码、配置、测试参数、脚本、Markdown、YAML、JSON、XML、SQL 和 properties 文件默认使用 UTF-8。
- 不得混用 GBK、GB2312、Latin-1、Windows-1252 等编码，除非目标项目已有明确兼容要求。
- 不得在同一文件中混用不同编码来源的文本。
- UTF-8 BOM 仅在项目现有工具链明确要求时使用；否则不主动新增 BOM。

## ENC-002: 禁止乱码和错误转码

生成内容 MUST 是人类可读文本，不得包含乱码。

- 注释、字符串常量、配置值、日志文案、错误消息、测试参数和文档说明都必须可读。
- 不得提交 replacement character、问号串、HTML 实体残留、URL 编码残留、双重转码文本或明显 mojibake 文本。
- 从外部材料复制文本后，必须确认编码正确，不得直接粘贴已损坏的字符。
- 如果原始需求、接口、配置或第三方返回值本身存在乱码，必须在 `context.md`、`design.md` 或 review 证据中记录来源和处理策略，不能把乱码当作正常标准文本继续扩散。

## ENC-003: 注释和代码文本

代码注释、Javadoc、docstring、错误消息和用户可见文本 MUST 使用正确编码。

- 中文注释必须能以 UTF-8 正确显示。
- 英文和技术标识符保持原文，不得被翻译或转码破坏。
- 字符串常量中的非 ASCII 文本必须是业务需要；否则优先使用清晰英文或项目默认语言。
- 不得通过手工拼接错误字节、错误 escape 或不可见控制字符来生成文本。

## ENC-004: 配置和资源文件

配置文件 MUST 使用项目工具链可正确读取的编码和转义方式。

- YAML、JSON、XML、properties、SQL、env template 和 shell/PowerShell 脚本必须可被对应解析器读取。
- properties 或 legacy 配置如果要求 ASCII escape，必须使用项目约定的 escape 工具或现有写法。
- 正则表达式、路径、URL、SQL、JSON 字符串和 shell 参数中的反斜杠、引号和换行必须按目标格式正确转义。
- 不得把显示层编码问题用业务配置硬编码绕过。

## ENC-005: 测试和验证

涉及非 ASCII 文本、配置、序列化、导入导出或跨系统传输的变更 MUST 验证编码。

- JSON 测试参数文件 SHOULD 覆盖中文、英文、符号、空白、换行和边界字符。
- API、CLI、文件导入导出、数据库读写、消息队列、日志和 UI 展示路径涉及文本时，必须验证输入输出不乱码。
- 生成文件必须能被项目工具链、测试框架、编译器、解释器、lint、formatter 或解析器正常读取。
- Review 证据必须记录编码/乱码检查结果，或者说明为什么本次变更不涉及文本编码风险。

## ENC-006: AI 生成内容要求

AI 生成或修改文件时 MUST 主动保护字符集。

- 编辑前应保留文件既有编码风格；新增文件默认 UTF-8。
- 如果工具输出显示疑似乱码，必须停下来重新读取源文件或用合适编码确认，不得继续基于错误显示修改。
- 不得把终端、网页、PDF、Office 文档或外部接口中已损坏的文本直接写入代码或配置。
- 完成前必须检查新增或修改内容中没有乱码、错误转义或不可读字符。

## ENC-007: Workflow 要求

- `design.md` MUST 说明本次变更是否涉及编码/字符集/非 ASCII 文本/配置转义风险。
- `tasks.md` MUST 为涉及文本、注释、配置或外部数据的任务列出编码和乱码验证要求。
- Alignment Review MUST 检查注释、代码、配置、测试参数和文档是否存在乱码。
- Security Review MUST 检查乱码修复是否引入敏感信息泄露、错误日志输出或错误配置。
- `/sp-complete` MUST 记录编码/乱码检查证据，或记录不适用原因。
