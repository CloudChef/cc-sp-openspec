# Business Standards

> 默认语言：除非用户明确要求英文，业务规范、功能说明和规则说明默认使用中文；代码标识符、API 路径、配置键和命令保持原文。

本文件用于沉淀项目级可复用业务规范。它不替代具体功能的 current-state feature docs，也不记录过程证据。

## 术语定义

- Source References / 来源引用：支撑业务定义、功能说明、流程说明、规则或需求结论的项目内来源，例如代码路径及其行为摘要、测试、配置、迁移/schema、API 文档、README/Wiki、部署文档或明确的用户确认。来源引用必须说明为什么能证明该结论，不用于粘贴代码或引入外部猜测。
- 待确认事项：当前来源不足、存在冲突、语义不清或必须由业务/项目负责人确认的行为。待确认事项不是已批准需求、规则、兼容承诺或实现决策。

## BUS-001: 业务说明边界

业务说明必须面向用户、客户、运营、系统行为和可观察结果。

- 功能说明、功能流程说明、业务定义、需求和场景不得直接出现代码、伪代码、调用链、SQL 片段、类名/方法名片段或条件表达式。
- 代码路径、类名、方法名、SQL、配置键和框架细节只允许出现在 Source References、Source Mapping、Design Baseline 或规则来源说明中。
- 不确定的业务意图必须记录为待确认事项，不得从实现细节推断成已确认规则。

## BUS-002: 当前功能业务定义

`/sp-code-to-spec` 生成 current-state feature docs 时，每个功能目录必须记录当前功能的业务定义。

- 文档识别层级是 product/project -> module -> feature -> function point：先在产品/项目中识别 module，再在 module 中识别 feature，最后在 feature 中识别 function point。
- 记录业务目的、参与者/客户端、领域对象、输入/输出、状态或生命周期术语、使用到的项目级业务规则。
- `<feature>.md` 记录功能目的、参与者/客户端、入口、功能点清单、当前行为概览和相关文档链接。
- `spec/spec.md` 记录 feature 级当前行为、业务规则、输入/输出、重要变体、错误/拒绝情况和可观察结果；多功能点 feature 必须在 `spec/` 下用真实功能点名拆分 `<function-point>-spec.md`。
- `design/design.md` 记录 feature 级当前技术实现、Source Mapping、共享类/模块职责、数据/配置/日志/安全/测试入口和实现约束；多功能点 feature 必须在 `design/` 下用真实功能点名拆分 `<function-point>-design.md`，并记录相关 class/module、low-level design、关键方法签名、参数定义、返回值、data flow、IO 和验证入口。
- `flow/flow.md` 记录 feature 级当前业务流程；多功能点 feature 必须在 `flow/` 下用真实功能点名拆分 `<function-point>-flow.md`，说明触发方式、前置条件、主流程、关键分支点、数据/状态变化、外部 IO 或异步行为，以及最终可观察结果。
- `other/` 仅用于补充说明，不作为必需过程证据目录。
- 单一功能的局部行为优先放在 `docs/<project-name>/<module>/<feature>/` 中；`/sp-code-to-spec` 不得把当前状态 feature docs 写入 `docs/standards/modules/` 或 `openspec/specs/`。
- `/sp-code-to-spec` 不要求旧式矩阵、流程或证据模板作为固定章节；只要求当前功能说明、当前行为规格、当前设计实现、功能点 low-level design 和当前流程说明能够支持后续维护。

## BUS-003: 项目级业务规则

只有稳定、可复用、跨功能或跨模块的业务规则才能写入本文件。

- 可写入的内容包括业务术语、领域对象定义、状态生命周期、权限/审批含义、准入/拒绝策略、计费/额度/优先级规则、数据归属和跨能力不变量。
- 规则必须说明适用范围、适用模块/功能、来源引用和待确认事项。
- 如果规则只来自单一可疑实现、遗留兼容逻辑、临时迁移逻辑或未确认推断，不得升级为项目级规则。
- 如果业务规则与现有 spec、wiki、代码或用户输入冲突，必须记录冲突并等待确认。

## BUS-004: 分支完整性

功能分支分析必须覆盖真实业务决策，而不是只记录入口路径。

- 对每个功能点列出参与者/触发方式、输入类别、前置状态、业务分支、输出/副作用、错误/拒绝原因和证据。
- 对状态迁移、审批、权限、准入、排序、评分、持久化、通知、异步和外部集成相关功能，必须记录对应分支或明确不存在。
- 不得把被更早校验、过滤或状态判断遮蔽的路径误认为已经证明目标业务分支。

## BUS-005: 功能流程完整性

功能流程说明必须让后续维护者不看代码也能理解业务路径。

- 每个功能点必须描述从开始/触发到最终可观察结果的完整业务流程。
- 流程说明必须覆盖主路径、关键分支点、数据/状态变化、外部 IO、异步行为和异常/拒绝结果中实际存在的部分。
- 流程说明必须引用 Source References，并把来源不足、冲突或需要负责人确认的部分放入待确认事项。
- 流程说明不得替代设计文档中的代码结构说明；代码路径、类名、方法名和调用链仍然只能放在 Source References、Source Mapping 或 Design Baseline 中。
